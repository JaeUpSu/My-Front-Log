
## Q.  SQL/DBMS/ORM 에 대한 설명을 해주세요?
<BR>

### &nbsp;&nbsp; `A. SQL => DB 와 상호작용하기 위한 언어`
### &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; `DBMS => DB 를 관리하고 조작하는 SW`

### &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; `ORM => 객체 지향 프로그래밍 언어와 관계형 데이터베이스 사이의 데이터 변환을 자동화하는 도구나 기술`

<BR>

```
 <🎙️ Answer >

@ SQL (Structured Query Language)

데이터베이스와 상호작용하기 위한 언어
데이터베이스에서 데이터를 저장, 검색, 수정 및 삭제하는 데 사용
SQL은 표준화된 구문과 명령어 집합을 제공
데이터베이스에서 테이블을 생성
쿼리를 실행하여 데이터를 필터링하고 정렬
데이터베이스 스키마를 변경하는 등의 작업을 수행 가능

@ DBMS (Database Management System)

데이터베이스를 관리하고 조작하는 소프트웨어 시스템
DBMS는 데이터베이스를 생성하고 유지 관리하며, 데이터를 저장하고 검색할 수 있는 인터페이스를 제공합니다. DBMS는 데이터베이스의 안정성, 보안, 일관성 등을 관리하며, 여러 사용자가 동시에 데이터베이스에 접근할 수 있도록 합니다.

@ ORM (Object-Relational Mapping)

객체 지향 프로그래밍 언어와 관계형 데이터베이스 사이의 데이터 변환을 자동화하는 도구나 기술
객체와 테이블 간의 매핑을 정의
개발자가 SQL 쿼리 대신 객체 지향적인 방식
데이터베이스를 다룰 수 있도록 도와줌 
데이터베이스 연산을 추상화하고 객체 지향적인 API를 제공
개발자가 데이터베이스와 상호작용하기 쉽게 함

```    


## Q.  Webpack을 통한 번들링 과정에 대한 이해?

<BR>

### &nbsp;&nbsp; `A. Webpack => 모듈 번들러`
### &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; `여러 개의 JavaScript 파일, CSS 파일, `

### &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; `이미지 등의 자원들을 하나로 묶어서 번들로 만들어주는 도구`

<BR>

```
 <🎙️ Answer >

Webpack을 사용함으로써 개발자는 모듈화된 개발 가능, 
여러 개의 파일을 하나의 번들로 묶어서 성능을 개선 가능

- 엔트리(Entry) 설정
    웹팩은 프로젝트의 시작점인 엔트리 포인트를 설정
    엔트리는 의존성 그래프의 시작점이며, 
    웹팩은 이를 기반으로 모든 종속 모듈을 찾아내어 번들링

- 의존성 그래프 생성
    웹팩은 엔트리 파일을 기준으로 의존하는 다른 모듈들의 종속성을 파악하여 의존성 그래프(Dependency Graph)를 생성
    이 그래프는 어떤 모듈이 어떤 모듈에 의존하는지를 나타내는 정보를 담고 있음

- 로더(Loader) 적용
    웹팩은 각 모듈을 로드하면서, 
    필요한 경우 로더를 통해 파일을 변환하거나 추가 처리를 수행 가능 
    예를 들어, Babel 로더를 사용하여 ES6+ 문법을 ES5로 변환하거나, CSS 로더를 사용하여 CSS 파일을 번들에 포함 가능

- 플러그인(Plugin) 적용
    웹팩은 로더 이외에도 플러그인을 통해 다양한 작업을 수행 가능 
    번들 최적화, 환경 변수 주입, 번들 결과물의 처리 등을 담당
    
    예를 들어, UglifyJsPlugin을 사용하여 JavaScript 코드를 압축, HtmlWebpackPlugin을 사용하여 HTML 파일을 생성 가능

- 번들 생성
    위의 과정을 거쳐 웹팩은 최종적으로 번들 파일(bundle)을 생성
    번들 파일은 하나의 JavaScript 파일
    HTML에서 로드되어 애플리케이션을 실행할 수 있는 형태
    번들 파일은 필요한 자원(이미지, 폰트 등)도 함께 포함 가능

```   



## Q.   Redux-toolkit과 SWR로 상태 관리를 설명?

<BR>

### &nbsp;&nbsp; `A. Redux Toolkit과 SWR은 프론트엔드에서`
### &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; `상태 관리를 용이하게 해주는 라이브러리`

<BR>

```
 <🎙️ Answer >

@ Redux Toolkit
    Redux 상태 관리 라이브러리를 보다 쉽고 
    간편하게 사용할 수 있도록 도와주는 도구 모음

    Redux의 핵심 개념인 액션, 리듀서, 스토어 등을 추상화
    보일러플레이트 코드를 최소화하여 
    Redux 개발의 생산성을 높임

    [주요 기능]

     - configureStore()
        Redux 스토어를 구성하는 간소화된 API 제공
     - createSlice()
        액션과 리듀서를 하나의 코드로 작성하고 자동으로 생성해줌
     - createAsyncThunk()
        비동기 작업을 처리하는 액션 생성자를 쉽게 생성해줌
     - createEntityAdapter()
        엔티티 관리를 위한 편리한 메소드 제공

    코드의 가독성과 유지보수성을 향상시키면서 
    Redux를 사용하여 상태 관리를 할 수 있도록 도와줌


@ SWR (Stale-While-Revalidate)
    데이터 요청 및 캐싱을 처리하기 위한 React 훅 라이브러리
    SWR은 원격 데이터를 가져와서 캐시하고, 화면에 표시한 후, 
    데이터를 백그라운드에서 다시 가져와서 갱신하는 방식으로 동작

    [주요 기능]
     - 자동 데이터 캐싱
        원격 데이터를 가져와서 자동으로 캐시하여 로컬에서 사용 가능

     - 자동 리로딩
        일정 시간 간격 또는 트리거에 따라 데이터를 자동으로 다시 가져와서 갱신

     - 에러 핸들링
        데이터 요청이 실패하는 경우 에러를 처리하고, 
        재시도 로직을 자동으로 처리 가능

     - 데이터 유효성 검사
        데이터의 유효성을 검사하여 최신 데이터로 갱신하고, 
        캐시된 데이터를 화면에 표시

    SWR은 서버 레이어의 API와 함께 사용되어 
    데이터 요청과 관련된 작업을 단순화하고, 
    렌더링 성능을 최적화할 수 있도록 도와줌
```   



## Q.   RESTful API에 대한 이해?

<BR>

### &nbsp;&nbsp; `A. Representational State Transfer의 약자로,`
### &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; `네트워크 기반의 아키텍처 스타일`

<BR>

```
 <🎙️ Answer >

웹 서비스를 설계하고 구현하는 데 사용되며, 
클라이언트와 서버 간의 통신을 위한 규칙과 원칙을 제공

Contents : 정의 / 주요 원칙 / 
           HTTP 메서드 역할 / 리소스 표현 어떻게
           상태 코드는 어떤 역할


Q. RESTful API란 무엇인가요?

A. RESTful API는 HTTP 기반의 웹 서비스 설계 원칙을 따르는 API
   리소스를 고유한 URI로 표현하고 HTTP 메서드를 사용하여
   리소스를 조작하는 방식

   이를 통해 클라이언트와 서버 간의 상호작용과 
   데이터 전송이 이룸



Q. RESTful API의 주요 원칙은 무엇인가요?

A. RESTful API의 주요 원칙은 다음과 같습니다:
        - Stateless (무상태성)
            각 요청은 독립적이며, 
            서버는 클라이언트의 상태를 유지하지 않습니다.

        - Client-Server 구조
            클라이언트와 서버는 독립적으로 개발
            서로의 변경 없이 동작 가능

        - Uniform Interface (통일된 인터페이스)
            리소스의 표현과 조작을 위한 
            통일된 인터페이스를 사용 필수
            
        - Cacheable (캐시 가능)
            응답은 캐시 가능해야 하며, 
            클라이언트는 캐시를 이용하여 성능을 향상 가능



Q. RESTful API에서 HTTP 메서드의 역할은 무엇인가요?

A. RESTful API에서 HTTP 메서드는 다음과 같은 역할을 수행합니다:
        GET: 리소스의 조회를 위해 사용합니다.
        POST: 새로운 리소스를 생성하기 위해 사용합니다.
        PUT: 기존 리소스를 업데이트하기 위해 사용합니다.
        DELETE: 리소스를 삭제하기 위해 사용합니다.



Q. RESTful API에서 리소스의 표현은 어떻게 이루어지나요?

A. RESTful API에서 리소스는 고유한 URI로 표현
   URI는 리소스의 위치와 식별자를 나타내며, 
   클라이언트는 이를 통해 특정 리소스에 접근하고 조작 가능




Q. RESTful API의 상태 코드는 어떤 역할을 수행하나요?

A. RESTful API의 상태 코드는 클라이언트에게 
   요청의 처리 상태를 알림 
   몇 가지 일반적인 상태 코드는 다음과 같습니다:
    200 OK: 성공적인 요청을 나타냅니다.
    201 Created: 새로운 리소스의 생성을 나타냅니다.
    400 Bad Request: 잘못된 요청을 나타냅니다.
    404 Not Found: 요청한 리소스를 찾을 수 없음을 나타냅니다.
    500 Internal Server Error: 서버 내부 오류를 나타냅니다



Q. HTTP 메서드인 PUT 과 PATCH 의 차이를 말해주세요?

A. 공통점은 모두 리소스의 업데이트를 위해 
   사용되는 메서드
   
   즉, PUT은 전체 리소스를 대체하거나 
   새로운 리소스를 생성하는 데 사용되며, 
   PATCH는 리소스의 일부를 수정하는 데 사용


```



## Q.   테스트 자동화에 대한 이해?

<BR>

### &nbsp;&nbsp; `A. 소프트웨어 개발에서 품질을 확보하고`
### &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; `버그를 줄이기 위해 자동화된 테스트를 실행하는 프로세스`

<BR>

```
 <🎙️ Answer >

Q. Frontend에서 테스트 자동화의 중요성은 무엇인가요?

A. Frontend에서 테스트 자동화는 애플리케이션의 품질을 보장
   개발자가 변경 사항을 안전하게 반영할 수 있도록 도와줌
   자동화된 테스트는 개발자의 생산성을 높이고 
   버그를 식별하여 조기에 해결할 수 있도록 함


Q. Frontend에서 어떤 종류의 테스트를 자동화할 수 있나요?

A. Frontend에서는 다양한 종류의 테스트를 자동화 가능 
   일반적으로는 단위 테스트, 통합 테스트, 기능 테스트, 
   UI 테스트 등이 포함 가능 
   
   단위 테스트는 개별 컴포넌트나 함수의 동작을 확인하며, 
   통합 테스트는 컴포넌트 간의 상호작용을 테스트
   기능 테스트는 전체 시스템의 기능을 테스트하고, 
   UI 테스트는 사용자 인터페이스의 동작을 확인
   
Q. Frontend 테스트 자동화를 위해 어떤 도구나 프레임워크를 사용할 수 있나요?

A. Frontend 테스트 자동화를 위해 다양한 도구와 프레임워크
   일반적으로 Jest, Mocha, Jasmine 등의 테스트 프레임워크와 
   함께 React Testing Library, Enzyme 
   등의 테스트 유틸리티를 사용 가능 
   
   또한, Cypress, Puppeteer, Selenium 등의 
   브라우저 자동화 도구를 사용하여 UI 테스트를 자동화 가능

   
Q. 테스트 커버리지란 무엇이며 왜 중요한가요?

A. 테스트 커버리지는 소프트웨어 테스트에서 얼마나 많은 
   코드가 테스트되었는지를 측정하는 지표 

   테스트 커버리지를 높이는 것은 애플리케이션의 안정성과 
   신뢰성을 높이는 데 도움
   
   커버리지가 낮으면 테스트되지 않은 코드 부분에 대한 
   버그나 예상치 못한 동작이 발생할 수 있기 때문에 중요
   

Q. CI/CD와 테스트 자동화는 어떤 관계가 있나요?

A. CI/CD (Continuous Integration/Continuous Deployment)는   
   애플리케이션의 지속적인 통합과 배포를 가능하게 하는 프로세스
   테스트 자동화는 CI/CD 파이프라인의 일부로 자동화된 테스트를 실행
   코드 변경 사항이 안정적인지 확인
   
   테스트 자동화는 CI/CD의 핵심 요소이며, 
   자동화된 테스트의 성공 여부에 따라 코드의 통합과 배포가 결정

```



## Q.   E-commerce 웹 개발에 대한 이해?

```
A. 성능 최적화를 위해 다음과 같은 접근 방식을 사용 가능

     - 캐싱
        반복적으로 호출되는 데이터나 페이지를 캐싱하여 
        성능을 향상
        
     - 이미지 최적화
        이미지 크기를 최적화하고 압축하여 
        로딩 시간을 단축

     - CDN (Content Delivery Network)
        전 세계에 분산된 서버를 통해 콘텐츠를 
        제공하여 로딩 속도를 개선
    
    - 코드 번들링과 압축
        Webpack 등의 도구를 사용하여 
        코드를 번들링하고 압축하여 
        파일 크기를 줄이고 로딩 속도를 향상

Q. 결제 처리를 위해 어떤 기술이 사용될 수 있나요?

A. 결제 처리를 위해 다양한 기술이 사용 가능
   일반적으로 신용카드 결제를 위해 
   Stripe, PayPal, Braintree 등의 
   결제 게이트웨이를 사용 가능 
   
   또한, 전자 지갑 서비스인 
   Apple Pay, Google Pay 등을 통한 모바일 결제 기능도 구현


   
Q. E-commerce 웹 개발에서 사용되는 인증과 권한 관리 방법은 무엇인가요?

A. E-commerce 웹 개발에서 사용되는 인증과 
   권한 관리 방법은 다양
   
   주로 사용자 계정을 생성하고 관리하기 위해 
   회원 가입 및 로그인 기능을 구현
   
   인증은 사용자가 신원을 확인하는 과정이며, 
   세션, 쿠키, 토큰 등을 사용하여 사용자를 식별
   
   권한 관리는 로그인한 사용자의 역할과 권한을 
   관리하여 접근 제어를 수행하는 과정
   
```





## Q.   PG 연동 개발에 대한 이해?

<BR>

### &nbsp;&nbsp; `A.  결제 처리를 위해 Payment Gateway (PG)`
### &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; `서비스와의 연동을 구현하는 것을 의미`

<BR>

```
 <🎙️ Answer >

Q. PG 연동 개발의 주요 단계는 무엇인가요?

A. PG 연동 개발의 주요 단계는 다음과 같습니다
    
    - PG 서비스 선택
        사용할 PG 서비스를 선택하고 계약을 체결합니다.

    - API 문서 확인
        선택한 PG 서비스의 API 문서를 확인하여 
        결제 처리에 필요한 엔드포인트, 파라미터, 
        응답 포맷 등을 파악
        
    - 결제 요청 구현
        웹 애플리케이션에서 PG 서비스로 
        결제 요청을 보내는 코드를 구현
        
    - 응답 처리
        PG 서비스로부터의 응답을 받아서 처리하고 
        결제 결과를 처리



Q. PG 연동을 위해 어떤 기술과 도구를 사용할 수 있나요?

A. PG 연동을 위해 다양한 기술과 도구를 사용할 수 있습니다. 
    
    주로 웹 애플리케이션의 백엔드 개발 언어와 
    프레임워크에 따라 HTTP 요청을 보내고 받을 수 있는 
    라이브러리를 사용
    
    또한, 암호화 기술과 보안 프로토콜을 사용하여 
    결제 정보를 안전하게 전송


Q. 결제 요청의 보안을 위해 어떤 접근 방식을 사용할 수 있나요?

A. - SSL/TLS 암호화
       결제 요청과 응답은 HTTPS 프로토콜을 사용하여 
       암호화된 연결을 통해 전송
    
   - 토큰화
       신용카드 정보를 저장하지 않고 
       대신 토큰을 사용하여 결제 요청을 처리 가능

   - PCI DSS 준수
       결제 카드 산업 데이터 보안 표준 
       (PCI DSS)을 준수하여 카드 데이터의 보안을 강화



Q. PG 연동 시 에러 핸들링은 어떻게 처리해야 하나요?

A. PG 연동 시 에러 핸들링은 중요한 부분
   에러 처리를 위해 다음과 같은 
   접근 방식을 사용 가능

        - 오류 메시지 처리
            PG 서비스로부터의 응답에 포함된 
            오류 메시지를 분석하여 사용자에게 
            적절한 안내를 제공
            
        - 예외 처리
            결제 요청 시 발생하는 예외 상황을 처리하고 
            적절한 예외 핸들링을 구현
            
        - 로깅
            에러 로그를 기록하여 추후 디버깅 및 분석에 
            활용 가능  



Q. PG 연동 시 테스트 환경에서의 테스트 접근 방식은 어떻게 될까요?

A. PG 연동 시 테스트 환경에서는 실제 PG 서비스와 
   연동하는 것이 아닌, 테스트용 PG 서비스나 
   시뮬레이터를 사용하여 테스트를 수행
   
   테스트 환경에서는 결제 성공, 실패, 취소 등 
   다양한 시나리오를 시뮬레이션하여 
   각 상황에 대한 처리를 검증

```



## Q.   Ramda, Lodash 등의 라이브러리 이해?

<BR>

### &nbsp;&nbsp; `A.  자바스크립트의 함수형 프로그래밍을`
### &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; `지원하는 유용한 유틸리티 라이브러리`

<BR>

```
 <🎙️ Answer >

데이터 조작, 함수 조합, 컬렉션 관리 등 
다양한 기능을 제공하여 개발자가 코드를 
더욱 간결하고 효율적으로 작성할 수 있도록 도움


Q. Ramda와 Lodash의 주요 특징은 무엇인가요?

A. Ramda와 Lodash는 모두 함수형 프로그래밍을 강조
   불변성을 지향하는 특징을 가지고 있습니다. 
   
   주요 특징은 다음과 같습니다

    - 함수 조합
        함수를 조합하고 연결하여 
        새로운 함수를 생성하는 기능을 제공

    - 커리(curry)
        함수를 부분적으로 적용하여 
        새로운 함수를 생성하는 커링을 지원
        
    - 컬렉션 조작
        배열이나 객체와 같은 컬렉션을 다루는 
        다양한 유틸리티 함수를 제공
        
    - 불변성
        데이터의 불변성을 유지하며, 
        순수 함수를 활용하여 부작용을 최소화
        
Q. Ramda와 Lodash의 차이점은 무엇인가요?

A. Ramda와 Lodash는 모두 유틸리티 함수를 
   제공하는 라이브러리이지만, 일부 차이점 존재

    - 함수 커리 기능
        Ramda는 기본적으로 모든 함수를 
        커링할 수 있도록 설계되어 있지만,
        Lodash는 특정 함수들만 커링 가능

    - 메서드 체이닝
        Lodash는 체이닝을 통해 연속적으로 
        함수를 호출하는 메서드 체이닝 기능을 제공
        Ramda는 체이닝을 지원하지 않습니다. 
        Ramda는 함수 조합을 위해 파이프라인 기법을 사용
        
    - 크기
        Lodash는 더 많은 유틸리티 함수를 포함하고 
        있어 더 큰 파일 크기를 가질 수 있음 
        Ramda는 작은 크기로 유지하려는 
        철학을 가지고 있음


Q. Ramda나 Lodash를 선택하는 기준은 무엇인가요?

A. Ramda와 Lodash는 개발자의 선호도와 
   프로젝트의 요구사항에 따라 선택 가능

    - 함수형 프로그래밍
        Ramda는 함수형 프로그래밍에 
        더 중점을 두는 라이브러리
        
        함수 조합과 커링 등 함수형 프로그래밍을 
        선호하는 경우에 적합
        
    - 메서드 체이닝
        Lodash는 메서드 체이닝을 통해 
        코드를 간결하게 작성 가능 
        
        다양한 유틸리티 함수를 사용할 수 있으므로, 
        이 기능을 활용하고자 하는 경우에 유용
        
    - 파일 크기
        프로젝트의 용량 제한이 있는 경우, 
        Ramda는 더 작은 파일 크기로 유지되므로 
        크기가 중요한 경우에 선택 가능
```
